# Context: Debug with Dependencies as Project References

## Purpose

This file defines rules for determining **which solution file to use** when building or debugging this repository, depending on whether local source-level debugging of dependencies is enabled.

---

## Solution Files

| File | When to use |
|---|---|
| `src/Telemetry.Samples.sln` | Default — all dependencies resolved from NuGet packages |
| `src/Telemetry.Samples.DebugDependencies.sln` | When one or more `*DirectImport` variables are `true` and the corresponding source repositories are available locally |

---

## Controlling Variables (per dependency family)

Each dependency family is controlled by a pair of MSBuild properties:

### Diginsight Core (`diginsight/telemetry`)

| Property | Default | Description |
|---|---|---|
| `DiginsightCoreSolutionDirectory` | _(empty)_ | Absolute path to the local clone of the `diginsight/telemetry` repository `src\` folder |
| `DiginsightCoreDirectImport` | `false` | Set to `true` to resolve Diginsight Core packages from local project references |

### Diginsight SmartCache (`diginsight/smartcache`)

| Property | Default | Description |
|---|---|---|
| `DiginsightSmartCacheSolutionDirectory` | _(empty)_ | Absolute path to the local clone of the `diginsight/smartcache` repository `src\` folder |
| `DiginsightSmartCacheDirectImport` | `false` | Set to `true` to resolve SmartCache packages from local project references |

---

## Where Variables Are Set

- **Defaults** are set in `src/Directory.Build.props`.
- **User overrides** go in `src/Directory.Build.props.user` (git-ignored, machine-local, created manually by each developer).

Example `Directory.Build.props.user`:

```xml
<Project>
  <PropertyGroup>
    <DiginsightCoreSolutionDirectory>E:\dev\Diginsight\telemetry\src\</DiginsightCoreSolutionDirectory>
    <DiginsightCoreDirectImport>true</DiginsightCoreDirectImport>
  </PropertyGroup>
</Project>
```

---

## Rules: Which Solution to Build

### Rule 1 — All DirectImport variables are false (or not set)

**→ Use `Telemetry.Samples.sln`**

This is the default and is always correct when `Directory.Build.props.user` does not exist or sets no `*DirectImport` to `true`.

### Rule 2 — Any DirectImport variable is true AND the source is available

**→ Use `Telemetry.Samples.DebugDependencies.sln`**

Prerequisites:
- The corresponding `*SolutionDirectory` is set and the path exists on disk.
- The `.csproj` files referenced from that directory are present (i.e., the dependency repository has been cloned and the `src\` subfolder is accessible).

### Rule 3 — Any DirectImport variable is true BUT the source is NOT available

**→ Cannot use `DebugDependencies.sln`**

Action required: either clone the missing dependency repository to the configured `*SolutionDirectory`, or set `*DirectImport=false` for the missing dependency and use `Telemetry.Samples.sln`.

---

## Decision Tree

```
Check Directory.Build.props.user (if it exists):

Is any *DirectImport == 'true'?
│
├── NO  ──► Build with: Telemetry.Samples.sln
│
└── YES ──► Does every *SolutionDirectory with DirectImport=true
            exist on disk with the expected .csproj files?
            │
            ├── YES ──► Build with: Telemetry.Samples.DebugDependencies.sln
            │
            └── NO  ──► Clone the missing repository, then use
                         Telemetry.Samples.DebugDependencies.sln
                        OR
                        Set DirectImport=false and use
                        Telemetry.Samples.sln
```

## Project File Convention

Every project that depends on a switchable dependency family must include both a conditional `ProjectReference` (for direct import) and a conditional `PackageReference` (for package-based import):

```xml
<!-- Project references: active only when DirectImport == true -->
<ProjectReference Include="$(DiginsightCoreSolutionDirectory)Diginsight.Core\Diginsight.Core.csproj"
                  Condition="'$(DiginsightCoreDirectImport)' == 'true'" />

<!-- Package references: active only when DirectImport != true -->
<PackageReference Include="Diginsight.Core" Version="$(DiginsightCoreVersion)"
                  Condition="'$(DiginsightCoreDirectImport)' != 'true'" />
```

Both the `ProjectReference` and `PackageReference` lines must be kept in sync when adding or removing dependency assemblies.

---

## CI / GitHub Actions

CI pipelines **always** use `Telemetry.Samples.sln`. The `DebugDependencies` solution is for local developer use only.  
The `Directory.Build.targets` file sets `ContinuousIntegrationBuild=true` when `GITHUB_ACTIONS=true`, which also disables any direct-import overrides that might accidentally reach the pipeline.

---

## Common Mistakes

| Mistake | Symptom | Fix |
|---|---|---|
| Opening `DebugDependencies.sln` without cloning dependency repos | Solution fails to load; project references unresolved | Clone the dependency repo or switch to `Telemetry.Samples.sln` |
| Toggling `DirectImport` without deleting `packages.lock.json` | Restore errors on `PackageReference` items | Delete all `packages.lock.json` files in `src\`, then restore |
| Editing `Directory.Build.props` instead of `Directory.Build.props.user` | Changes committed to source control, breaking CI for others | Move overrides to the `.user` file |
| Building `Telemetry.Samples.sln` while `DirectImport=true` | Compiler resolves both `ProjectReference` AND `PackageReference` (duplicate symbols) | Use `DebugDependencies.sln` when `DirectImport=true` |
